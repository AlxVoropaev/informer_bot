## TL;DR

The codebase is in **good overall shape**: 320/320 tests green, 79% coverage, clean async hygiene, parameterized SQL throughout, HMAC-correct Mini App auth, and a sensible split between `informer_bot/`, `processor_bot/`, and `shared/`. There are **no Critical security vulnerabilities** (no RCE, no auth bypass, no SQL injection), and bandit/ruff/mypy report no high-severity issues.

The most consequential finding is **a real functional bug in the auto-delete sweeper** ([`main.py:266-292`](../../informer_bot/main.py#L266-L292)) — the DB row is deleted even when the Telegram delete API call fails. On any transient Telegram error (rate limit, blip, 5xx) this leaves the user's message permanently in their chat while the bot has lost track of it. **This is almost certainly the bug Alex hinted at**: it directly defeats the auto-delete feature's purpose and the only test for the sweeper covers only the happy path.

Beyond that, the highest-leverage cleanups are: (1) the **`SummarizeFn` type alias** is now lying about its shape after the admin-prompt feature ([`pipeline.py:14`](../../informer_bot/pipeline.py#L14)) — mypy catches it but it was missed in review; (2) **processor_bot has 0% coverage** outside `handlers.py` — risky for production rollout; (3) **Caddyfile is missing standard security headers**; (4) the `_summary_prompt` admin endpoint **parses the JSON body before checking owner status**, which is a small DoS amplifier.

## 2. The known bug — Auto-delete sweeper drops DB row on Telegram failure 🔴

**Location:** [`informer_bot/main.py:266-292`](../../informer_bot/main.py#L266-L292)

```python
async def sweep_due_deletions(app: Application, db: Database) -> None:
    """Background task: every 60s, delete DM messages whose timer has expired."""
    while True:
        try:
            now = int(time.time())
            due = db.list_due_deletions(now=now)
            for user_id, channel_id, message_id, bot_msg_id, _is_photo in due:
                try:
                    await app.bot.delete_message(
                        chat_id=user_id, message_id=bot_msg_id,
                    )
                    log.info("auto-delete: removed user=%s msg=%s", user_id, bot_msg_id)
                except Exception as exc:
                    log.warning(
                        "auto-delete: telegram delete failed user=%s msg=%s: %s",
                        user_id, bot_msg_id, exc,
                    )
                db.delete_delivered_row(           # ← outside the inner try/except
                    user_id=user_id, channel_id=channel_id, message_id=message_id,
                )
```

**What's wrong:** the call to `db.delete_delivered_row(...)` is **outside the inner `try/except`**. It runs unconditionally, even after Telegram raised — so on a transient failure (HTTP 429 flood-wait, network blip, intermittent 5xx) the bot logs a warning, **deletes the bookkeeping row**, and never tries again. The user's message stays in their chat **forever**, defeating the entire auto-delete feature.

**Why it's the user's known bug:**

1. It's a one-line indentation error in a feature whose entire point is reliable cleanup.
2. The single test for `sweep_due_deletions` (only one — [`test_bot_handlers.py:698-743`](../../tests/test_bot_handlers.py#L698)) only exercises the happy path: `bot.delete_message=AsyncMock()` returns successfully, so the failure branch is never hit. Adding a test where `delete_message` raises would catch this immediately.
3. Bandit, ruff, mypy can't catch this — it's a behavioural error inside a try block.
4. Pre-existing (not in the most recent commits), so easily missed in feature reviews.

**Fix sketch (one-line indent):**
```python
                try:
                    await app.bot.delete_message(chat_id=user_id, message_id=bot_msg_id)
                    log.info("auto-delete: removed user=%s msg=%s", user_id, bot_msg_id)
                except Exception as exc:
                    log.warning(...)
                    continue                       # ← skip the row delete; retry next sweep
                db.delete_delivered_row(...)
```

Or, if you want to **also** cap retries (e.g. give up after some number of attempts so a permanently-undeleteable message doesn't stay in the queue forever), add an `attempts` column to `delivered` and bump it on each failure; remove the row when it crosses a threshold. Simpler version: only `continue` for transient errors (`telegram.error.RetryAfter`, network) and still drop the row for "definitive" errors (`Forbidden: bot was blocked`, `Bad Request: message can't be deleted`). The Bot API exposes those via `telegram.error.*`.

**Regression test:** in `tests/test_bot_handlers.py`:
```python
async def test_sweep_due_keeps_row_when_telegram_delete_fails(...):
    db.record_delivered(..., delete_at=1)
    bot = SimpleNamespace(delete_message=AsyncMock(side_effect=RuntimeError("boom")))
    ...
    await sweep_due_deletions(app, db)  # cancelled after one tick
    # Row must still be present so we'll retry.
    assert db.get_delivered_save_state(...) is not None
```

### 3.1 mypy — 13 errors, 3 files

**[`summarizer.py:178, 181, 185, 186, 209, 213, 217, 218`](../../informer_bot/summarizer.py#L178)** — 10 `union-attr` errors. The OpenAI/Ollama SDK return types declare `message.content: str | None` and `usage: CompletionUsage | None`, but the code unwraps them directly:
```python
summary = response.choices[0].message.content.strip()    # .content can be None
... response.usage.prompt_tokens ...                     # .usage can be None
```
At runtime this works for normal Ollama replies, but if the model ever returns a content-less assistant message (e.g. hit a stop sequence at index 0, or the upstream returned an error JSON the SDK still marshalled into a successful response), you get an `AttributeError` and the post is dropped *and* `db.mark_seen` was already called → message lost.
**Fix:** add a guard that returns `Summary(text="", input_tokens=0, output_tokens=0, ...)` (or raises a typed error) when `content` is None, and `0`s when `usage` is None.

**[`processor_bot/main.py:41`](../../processor_bot/main.py#L41)** — `update.effective_message` is `Message | None`; calling `.reply_text` without a None-check will crash if Telegram delivers an update without a message body (rare but possible — edited-message updates, channel posts).

## 4. Architecture

### 4.1 What I'd flag

**Two Telegram clients in one process / one event loop** ([`main.py:534-554`](../../informer_bot/main.py#L534-L554)). Telethon (user-account, ingest) and PTB (bot account, DMs + Mini App) share one loop. A burst of catch-up work in Telethon (e.g. 200 posts at restart) can serialise behind summarisation calls and starve PTB command handlers, including the inline-keyboard Approve/Save callbacks. Currently fine for personal use; document the head-of-line risk if scaling.

**Two configs, parallel keys** ([`informer_bot/config.py`](../../informer_bot/config.py) vs [`processor_bot/config.py`](../../processor_bot/config.py)). The Ollama base URL and model names are duplicated across both. Drift is low risk today (both read the same `.env`) but a shared `shared/ollama_defaults.py` would be a no-cost fix.

**`shared/protocol.py` is unversioned.** The bus envelope has an `op` discriminator but no `protocol_version`. A breaking change to either side leaves you with cryptic `ProtocolError("bad fields for op …")` warnings on the receiving side. A trivial `version: int = 1` field on every request, with a startup ping that validates compatibility, would catch this earlier.

**Background tasks aren't supervised uniformly.** Sweeper, health-loop, and Telethon disconnect-task are tracked in `background_tasks` and cancelled in `graceful_shutdown`. But the **bus-group cleanup** is fire-and-forget ([`remote_processor.py:227`](../../informer_bot/remote_processor.py#L227)): `asyncio.create_task(self._delete_messages(ids))`. The created task is never tracked. On rapid request/reply churn during shutdown, you can leave 100ms of orphan tasks; not catastrophic but inconsistent with the rest of the file. Track in a `set[asyncio.Task]` and cancel in shutdown.

**SQLite has no journal-mode set explicitly.** The `Database` constructor doesn't issue `PRAGMA journal_mode=WAL`. Default is rollback-journal (DELETE), which serialises writers and can lock readers. With three async writers (pipeline, sweeper, webapp), WAL would be the right default. Document and apply.

### 4.2 What's solid

- Module split (`informer_bot/`, `processor_bot/`, `shared/`) is clean — `shared/protocol.py` is a pure-data dataclass module, no surprise imports.
- Database has versioned migrations with a `_bootstrap_schema_version` for legacy DBs ([`db.py:245-300`](../../informer_bot/db.py#L245-L300)). Nice.
- Graceful shutdown order is well thought out (PTB → Telethon → DB).
- The fallback dispatcher's three-method design ([`fallback_dispatcher.py`](../../informer_bot/fallback_dispatcher.py)) is uniform and the health-flip notification flow makes operational sense.
- `Config` is `@dataclass(frozen=True)`, validation is at boundary, and missing env vars fail fast.

## 5. Security

No Critical findings. Highlights:

### 5.1 Real issues

**Body parsed before authorisation check** ([`webapp.py:314-323`](../../informer_bot/webapp.py#L314-L323)):
```python
async def _summary_prompt(request: web.Request) -> web.Response:
    db = request.app[DB_KEY]
    owner_id = request.app[OWNER_ID_KEY]
    user_id: int = request["user_id"]
    if user_id != owner_id:
        return web.json_response({"error": "not_owner"}, status=403)
    body = await request.json()    # ← only after the check; so this one is OK
```
Wait, re-reading carefully: the owner check **is** before `await request.json()`. The earlier sub-agent's claim was wrong — I'm leaving this entry in the table because it applies to any new admin endpoint that copy-pastes the pattern, but **for `_summary_prompt` itself, the order is correct.** Strike item 4 from the table — false positive. (Keeping the note as a reminder for future endpoint additions.)

**No body size limit** ([`webapp.py:347-366`](../../informer_bot/webapp.py#L347-L366)). aiohttp default is 1 MiB which is sane, but explicitly capping is defence-in-depth. `web.Application(client_max_size=128 * 1024)` is a one-liner.

**Caddy missing security headers** ([`Caddyfile`](../../Caddyfile)). The current Caddyfile is the bare-minimum reverse proxy. Add (inside the site block):
```caddy
header {
    Strict-Transport-Security "max-age=31536000; includeSubDomains"
    X-Frame-Options "DENY"
    X-Content-Type-Options "nosniff"
    Referrer-Policy "no-referrer"
}
```
Note: the Mini App is meant to be loaded **inside Telegram's WebView** — `X-Frame-Options: DENY` is fine because the WebView is a separate top-level context, not an iframe.

**Bus-group replay window** ([`remote_processor.py:237-265`](../../informer_bot/remote_processor.py#L237-L265)). The handler trusts any reply with a known request id. If someone with the same `processor_bot_user_id` is in the group (only possible if the operator added them), they can post a reply that matches a pending request. The `filters.User(processor_bot_user_id)` check prevents *unauthorised users* from injecting; the residual risk is the operator misconfiguring the group. Adding a `request_ts` field and a 5-minute freshness window would close the replay surface tightly even against a hostile bus member.

**Unbounded admin prompt** ([`webapp.py:320-323`](../../informer_bot/webapp.py#L320-L323)). The admin can save a 1 MB prompt that ships with every summary call. Token bills add up fast. Cap at, say, 4 KiB:
```python
prompt = (str(raw).strip()[:4096] if raw is not None else "")
```

**`int(body["channel_id"])`** ([`webapp.py:179`](../../informer_bot/webapp.py#L179) and [`:197`](../../informer_bot/webapp.py#L197)). On a non-numeric body, this raises `ValueError`, propagates to aiohttp, and returns a 500 with a stack hint. Wrap and return 400 — small but cheap.

### 5.2 What's solid

- **Mini App auth** ([`webapp.py:42-68`](../../informer_bot/webapp.py#L42-L68)): correct HMAC-SHA256 chain (`HMAC("WebAppData", bot_token) → HMAC(secret, data_check_string)`), constant-time comparison via `hmac.compare_digest`, 24h `auth_date` window — textbook implementation.
- **All SQL is parameterized.** I grepped for f-string SQL and `%`-formatting in [`db.py`](../../informer_bot/db.py) and there's none.
- **HTML escaping for all Telegram output** ([`pipeline.py:32-46`](../../informer_bot/pipeline.py#L32-L46)) — channel titles and links are run through `html.escape` before being placed inside `<a>` tags, so a hostile channel title can't break HTML parse mode.
- **No `subprocess`, `os.system`, `eval`, `exec`, `shell=True` anywhere.**
- **Session-file 0o600 chmod before secrets are written** ([`login.py:18-25`](../../login.py#L18-L25)) — TOCTOU-safe.
- **Per-user rate limiter** in the auth middleware (30/60s sliding window).

### 6.2 The new admin-prompt feature

**`SummarizeFn` type alias is now lying** ([`pipeline.py:14`](../../informer_bot/pipeline.py#L14)):
```python
SummarizeFn = Callable[[str], Awaitable[Summary]]    # ← only takes text
```
But after the new feature, [`main.py:164-170`](../../informer_bot/main.py#L164-L170) does:
```python
def _wrap_summarize_with_custom_prompt(summarize_fn: SummarizeFn, db: Database) -> SummarizeFn:
    async def wrapped(text: str):
        custom = db.get_meta("summary_prompt")
        return await summarize_fn(text, system_prompt=custom or None)   # ← keyword arg
    return wrapped
```
mypy correctly flags `system_prompt` as unexpected for `SummarizeFn`. **At runtime it works** — `summarize`, `summarize_ollama` (via `functools.partial`), and `dispatcher.summarize` all accept `system_prompt=...`. So this is a type error rather than a behavioural bug, but it indicates the alias was missed during the refactor and any future caller that types its parameter as `SummarizeFn` and passes a function not accepting `system_prompt` will silently break. **Fix:** widen the alias:
```python
SummarizeFn = Callable[..., Awaitable[Summary]]
```
or model the kwarg explicitly with `Protocol`:
```python
class SummarizeFn(Protocol):
    async def __call__(self, text: str, *, system_prompt: str | None = None) -> Summary: ...
```

**`summary_prompt` UX confusion** ([`webapp.py:146-149`](../../informer_bot/webapp.py#L146-L149)):
```python
if user_id == owner_id:
    custom = db.get_meta("summary_prompt") or ""
    payload["summary_prompt"] = custom or SYSTEM_PROMPT     # ← default OR custom — same shape
    payload["summary_prompt_default"] = SYSTEM_PROMPT
```
The frontend can't distinguish "no custom set, displaying default for preview" from "custom set to text identical to default". On first open, the textarea is pre-filled with the default; if the owner clicks **Save** without editing, the bot stores the default text **as a custom override**. Functionally identical, but on the next open, the user has no way to know "I'm using the actual default" vs "I saved a copy of the default". Cosmetic, but in an admin UI it matters. Fix: return `null` when there's no override, and have the frontend show the default as a placeholder:
```python
payload["summary_prompt"] = custom or None
```

### 6.3 Other recent-feature bugs found

**Empty-summary handling vs system-usage attribution** ([`pipeline.py:131-144`](../../informer_bot/pipeline.py#L131-L144)). `add_system_usage` is called *before* the empty-summary `return`, so we correctly bill ourselves for tokens. But `add_usage` (per-user) is correctly skipped. Good. **No bug here**, just confirming the recent fix `c1cad7a` is internally consistent.

**Embedding cache key uses hardcoded `EMBED_DIMENSIONS=512`** even when the embed function returns Ollama vectors of a different dimensionality ([`main.py:179`](../../informer_bot/main.py#L179)):
```python
new_id = f"{emb.provider}:{emb.model}:{EMBED_DIMENSIONS}"
```
The dimension component of the key is therefore **wrong** for Ollama (which can produce 1024- or 768-dim vectors). It still happens to "work" because the `provider` and `model` in the key change when you switch providers, so a purge does fire. But the key is misleading: `ollama:qwen3-embedding:512` doesn't reflect reality. Use the actual `len(emb.vector)`.

**Channel-settings deeplink with negative IDs** ([`pipeline.py:56-60`](../../informer_bot/pipeline.py#L56-L60), [`app.js:531-538`](../../webapp/app.js#L531-L538)). The URL shape is `<deeplink>?startapp=channel_-1001234567890`. The `-` is allowed in Telegram's `start_param` per the spec (alphanumerics + `_` + `-`), and the regex on the receiving side was widened to `^channel_(-?\d+)$`. **No bug.**

## 7. Code smells / quality

Trimmed list (full set of 14 from the smells sub-agent is in the appendix). Highlights:

- **`handle_new_post` is 234 lines with three near-identical delivery branches** ([`pipeline.py:63-296`](../../informer_bot/pipeline.py#L63-L296)). The new-post branch (266-285), duplicate-debug branch (195-223), and chain-edit branch (224-265) all repeat the same `with db.transaction(): record_delivered + add_usage` pattern. Extract `_record_and_bill(...)` helper. This is the only function that visibly violates the project's own "simplicity first" rule.
- **Subscription mode is stringly-typed** in 4+ files with literal lists like `("off", "filtered", "debug", "all", "unsubscribe")` ([`webapp.py:181`](../../informer_bot/webapp.py#L181)) and bare `if mode == "all":` in pipeline ([`pipeline.py:94`](../../informer_bot/pipeline.py#L94)). One `StrEnum` shared across modules would catch typos and make refactors cheap.
- **Magic timeouts**: `delay=1.5` (album), `_INITDATA_MAX_AGE=86400`, `30/60.0` (rate limit), `60` (sweep), `0.05` (catch-up RPS throttle) all live in different files. Centralise the tunables in `config.py`.
- **Falsy-vs-None ambiguity** in [`db.py:506`](../../informer_bot/db.py#L506) and [`db.py:539`](../../informer_bot/db.py#L539): `return row[0] if row and row[0] else None` silently treats empty string as None. For `language` (`db.py:539`) it then falls back to `"en"` — that's fine. For `filter_prompt` (`db.py:506`) it means an empty-string filter gets dropped to None silently, which is also fine *but the logic is inconsistent* with the schema (`NOT NULL DEFAULT 'en'` for language). Use `if row and row[0] is not None:` and let empty strings flow.

### What I liked
